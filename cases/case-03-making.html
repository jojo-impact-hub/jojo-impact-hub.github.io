案 3 製作說明

郵件寄送登記系統

Claude Code Apps Script Google Sheets GitHub Pages 六階段流程 3 個踩雷紀錄

從「換台電腦就看不到紀錄」出發，用 Claude Code 協作，最終做出一套跨裝置、有密碼保護、資料存在 Google 試算表的合約寄送登記系統。單一 HTML 檔案，無需安裝，打開連結即可使用。

背景 製作流程 指令範本 踩雷紀錄 成品特色 心得提醒

01 背景與痛點

痛點 1

原本的登記系統把資料存在 localStorage（瀏覽器本機），換台電腦就完全看不到之前的紀錄。

痛點 2

辦公室有三台電腦加一支手機，合約寄送是業務核心記錄，需要隨時在任何裝置上查得到。

痛點 3

廠商和窗口欄位是兩個分開的下拉，每次都要分別選，且不會自動帶入對方的 email 和電話。

痛點 4

合約寄出後要追蹤「對方是否已用印」，但原本沒有這個欄位，每次要另外記在別的地方。

02 六階段製作流程

1
功能需求確認
把廠商和窗口合併成單一下拉欄位（格式：「廠商名稱 ／ 窗口姓名」），選完自動帶入對方的 email 和電話。同時新增「對方用印狀態」追蹤欄位（無需用印 ／ 待用印 ／ 已用印），以及窗口電話欄位。每項需求先說清楚，讓 Claude 確認可行後才動工。
2
選擇跨裝置架構
說明有多台裝置的問題，請 Claude 評估解法。Claude 列出幾個選項：Firebase（需要開帳號）、GitHub Gist（讀取方便但寫入麻煩）、Google Drive API（設定複雜）、Google Apps Script + Google Sheets（已熟悉、資料在試算表直接看得到、免費）。選了最熟悉的 Apps Script 方案。
3
建立 Apps Script 後端
在 Google 試算表的「擴充功能 → Apps Script」建立專案，Claude 寫出 doGet（讀取寄送紀錄與預設廠商清單）和 doPost（新增 ／ 更新 ／ 刪除紀錄、儲存廠商）兩個函式。部署為 Web App（執行身分：我；誰可以存取：所有人），取得 API 網址。
4
HTML 前端串接 API
把原本讀寫 localStorage 的所有操作，改成用 fetch() 呼叫 Apps Script API。新增讀取中的 loading bar，讓使用者知道資料正在載入。廠商管理 Modal 升級為每筆含四個欄位（廠商名稱、窗口姓名、Email、電話），選廠商時自動帶入對應資訊。
5
上傳 GitHub Pages 解決跨裝置問題
本機用 file:// 開 HTML 檔案，瀏覽器的安全限制（CORS）可能導致 API 呼叫失敗。解決方式是把 HTML 上傳到 GitHub Pages——有真實的 HTTPS 網址，CORS 正常，任何裝置打開同一個網址就能存取同一份資料。
6
加入密碼保護
GitHub Pages 預設是公開的，合約寄送記錄不宜完全公開。在頁面最前面加一個全螢幕密碼覆蓋層，輸入正確才載入資料。同一個 session 內不需要重複輸入。密碼存在 HTML 第一行的常數，之後要改直接改那個字就好。

03 指令範本（可複製套用）

指令 1 對應階段 2｜說明跨裝置問題，請 Claude 評估架構
「這個 HTML 現在把資料存在 localStorage，但我有三台電腦，換台電腦就看不到紀錄了。我需要資料可以跨裝置同步，有什麼方法可以做到？幫我評估幾個選項的難易度。」
為什麼這樣說：先描述你的現實狀況（幾台裝置、現在的問題），再請 Claude 評估選項，而不是直接問「怎麼接 Firebase」——因為你可能還不知道 Firebase 適不適合，讓 Claude 幫你比較再做決定。
指令 2 對應階段 3｜請 Claude 建立 Apps Script 後端
「好，我選 Google Apps Script + Google Sheets。幫我寫 Apps Script 的程式碼，要能讀取寄送紀錄、新增一筆、刪除一筆、更新用印狀態，還有讀取和儲存常用廠商清單。部署方式也說明一下。」
為什麼這樣說：決定架構後，把所有需要的 API 功能一次說完，Claude 就能寫出完整的程式碼，不需要一個功能一個功能慢慢加，也不會遺漏。
指令 3 對應階段 4｜升級前端串接 API
「Apps Script 部署好了，URL 是〔貼上你的 URL〕。現在幫我把 HTML 裡面所有讀寫 localStorage 的地方，都改成呼叫這個 API。讀取的時候加個 loading 提示，讓我知道資料正在載入。」
為什麼這樣說：直接把 API URL 提供給 Claude，並說明要把哪些行為對接過去。加「loading 提示」這種 UX 細節也一起說，Claude 就會一併處理，不用後來再補。
指令 4 對應階段 6｜加入密碼保護
「這個頁面要放在 GitHub Pages，但資料是我們公司的合約寄送紀錄，不想讓不相關的人看到。可以加一個密碼保護嗎？輸入正確密碼才能看到內容，同一次開著瀏覽器不需要重複輸入。」
為什麼這樣說：說清楚為什麼需要（公開但不想公開）和體驗需求（同 session 不重複輸入），Claude 會選用 sessionStorage 實作，既達到目的又不麻煩。

04 踩雷紀錄

雷 1 Apps Script 首次部署跳出「未驗證應用程式」警告
狀況
點「授權存取」後，Google 跳出警告頁「Google 尚未驗證這個應用程式」，以為是做錯了或有安全問題，不敢繼續。
修法
這是正常的。點「進階」→「前往〔專案名稱〕（不安全）」→ 繼續授權，就能正常部署。因為這是你自己寫給自己用的 Script，Google 不會主動驗證，出現這個畫面代表流程沒有問題。
雷 2 廠商格式升級後，Apps Script 也要同步更新並重新部署
狀況
把廠商預設格式從兩個欄位（廠商、窗口）升級成四個欄位（廠商、窗口、Email、電話）後，前端正常但讀取資料卻出錯，或新欄位存不進去。
修法
Apps Script 的 doGet 和 doPost 都要同步修改來支援新格式，然後重新「部署 → 管理部署」→ 建立新版本。光改程式碼不夠，一定要重新部署才會生效。
雷 3 廠商清單出現「company／contact」假資料
狀況
系統開起來後，廠商下拉清單出現「company」「contact」等英文假資料，不是自己輸入的。
修法
是 Google Sheets 的欄位標題列被讀進來了。到 Google 試算表的「廠商窗口」分頁，把第一列的欄位名稱（company、contact 等）手動刪掉，或確認 Apps Script 的讀取範圍從第二列開始（跳過標題列），重新載入就正常。

05 成品特色

廠商＋窗口合併下拉，格式統一好辨識
選廠商自動帶入 email 和電話，不用手動查
對方用印狀態三階段追蹤（無需 ／ 待用印 ／ 已用印）
資料存在 Google Sheets，三台電腦 ＋ 手機都同步
前端密碼保護，不對外公開資料
CSV 一鍵匯出，方便備份或整理
廠商管理 Modal，隨時新增 ／ 修改 ／ 刪除廠商資料
單一 HTML 檔案，無需安裝，打開網址即可使用

06 心得提醒

01 localStorage 只存在那台電腦的那個瀏覽器——資料要跨裝置，就一定需要真正的後端。Google Sheets 就是最好找的後端。
02 Apps Script 出現「未驗證應用程式」不是錯誤，是正常流程。自己寫給自己用的 Script 都會跳這個，點進階繼續就好。
03 改了 Apps Script 程式碼，一定要重新部署（建立新版本）才會生效，光儲存是不夠的。
04 把 HTML 放到 GitHub Pages 不只是為了分享，也是為了解決 CORS 問題——本機 file:// 開啟可能擋住 API 呼叫。
05 一個 HTML 檔案 ＋ Apps Script ＋ Google Sheets，就能做到「資料庫 ＋ 跨裝置 ＋ 密碼保護」，不需要後端工程師，不需要付費服務。
